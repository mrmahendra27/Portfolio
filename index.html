Event-Driven Workflow Platform

Private · Entrata

Problem

Critical operational alerts depended on synchronous backend execution paths and tightly coupled retry logic, making delayed retries, repeated escalations, and long-running workflows difficult to manage reliably under failure conditions.

Constraint

The system required configurable recurring notifications with strict idempotency guarantees, resilient retry behavior, and fault isolation between workflow stages. Failures in downstream processing could not block upstream application flows or create duplicate notification delivery.

Architecture

Designed and implemented an event-driven workflow orchestration platform using RabbitMQ, AWS EventBridge, and Amazon SQS to support delayed execution, recurring processing, and distributed retry coordination across asynchronous services.

Built queue-backed orchestration pipelines capable of continuously scheduling and re-queuing events at fixed intervals until acknowledgment conditions or expiration windows were satisfied. Introduced asynchronous execution boundaries between processing stages to isolate failures, improve recovery behavior, and reduce operational pressure on synchronous application paths.

Integrated multi-channel notification delivery pipelines using Twilio, FCM, and APNs to support scalable SMS and mobile push notification workflows across distributed backend services. Implemented deterministic retry handling, idempotent processing patterns, and explicit failure recovery mechanisms to maintain consistency under partial failure scenarios.

Impact

  • Improved reliability of long-running asynchronous workflows
  • Eliminated blocking retry logic from synchronous application flows
  • Reduced duplicate event processing through idempotent workflow execution
  • Increased operational resiliency through distributed failure isolation
  • Established scalable event-driven processing patterns across backend services
PHP RabbitMQ AWS SQS AWS EventBridge Twilio FCM APNs PostgreSQL

Smart Maintenance Assistant

Private · Entrata

Problem

Operational support workflows relied on repetitive form-based intake processes and manual triage, creating delays in issue resolution, fragmented context gathering, and inefficient routing of operational tasks.

Constraint

The platform needed to leverage AI for conversational intake, classification, and recommendation generation without allowing non-deterministic model output to directly impact critical backend workflows. All downstream orchestration had to remain observable, recoverable, and auditable.

Architecture

Built an AI-powered conversational operations platform using Amazon Bedrock, LiteLLM, and retrieval-based architectures to transform structured intake workflows into adaptive, context-aware interactions.

Designed retrieval pipelines that analyzed incoming queries against historical operational data, prior resolution patterns, and internal knowledge sources to provide guided troubleshooting, dynamically request additional contextual information, and reduce unnecessary escalation into backend systems.

Implemented asynchronous orchestration pipelines using AWS SQS to transition unresolved interactions into structured backend workflows enriched with conversational metadata and operational context. Combined LLM-driven reasoning with deterministic queue-backed processing to ensure backend execution remained fault-tolerant, observable, and operationally safe.

Introduced modular AI orchestration layers to separate conversational reasoning, retrieval logic, workflow execution, and downstream system integration, improving maintainability and scalability across the platform.

Impact

  • Reduced manual operational triage through AI-assisted conversational workflows
  • Improved contextual data collection during intake and escalation flows
  • Increased operational efficiency by minimizing unnecessary backend escalations
  • Established reliable AI-to-workflow orchestration boundaries for safe production execution
  • Built scalable retrieval and orchestration pipelines for future AI-driven operational automation
Amazon Bedrock LiteLLM RAG Python AWS SQS PostgreSQL

Distributed Data Intelligence Platform

Private · WeAssemble

Problem

Data-intensive ingestion and analytics workflows processing external datasets were bottlenecked by synchronous scraping pipelines, inefficient database access patterns, and inconsistent legacy data behavior, resulting in slow reporting generation and reduced downstream performance.

Constraint

The platform needed to significantly increase ingestion throughput while preserving data quality, supporting concurrent processing, and maintaining predictable ETL behavior across continuously changing external data sources.

Architecture

Designed and implemented a distributed data ingestion and analytics platform using concurrent scraping and asynchronous processing patterns. Built scalable scraping and automation pipelines using Python and Selenium to extract structured and semi-structured data from dynamic web sources.

Developed ETL workflows using Pandas for large-scale cleaning, normalization, enrichment, and aggregation before persisting optimized analytical datasets into PostgreSQL. Introduced query optimization and indexing strategies to improve read-heavy analytical workloads and reduce reporting latency.

Built high-performance API layers using FastAPI to expose processed datasets and analytical insights while supporting scalable concurrent access patterns. Generated visualization and reporting pipelines using Matplotlib for downstream analytics consumption.

Impact

  • Improved scraping throughput by 50%
  • Reduced database and reporting latency by 35%
  • Increased reliability of large-scale ingestion workflows
  • Established scalable ETL pipelines for predictable analytics processing
